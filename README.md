## 🔧 新しいスクリプト構成

### 概要
リファクタリングにより、スクリプトを以下の5つのファイルに分割しました：
- **config.gs**: 設定の一元管理
- **commonLib.gs**: 共通関数ライブラリ
- **ApprovalFunctionR61_pjA.gs**: メイン承認ワークフロー処理
- **initSheetR1_pjA.gs**: シート初期化処理
- **sendFormR2_pjA.gs**: フォーム送信時のメール処理

### 各ファイルの機能

#### config.gs (設定ファイル)
- **目的**: 全スクリプトで使用する設定値を一元管理
- **主な設定内容**:
  - スプレッドシートID、シートGID
  - 承認ワークフローの列番号設定
  - メール設定（宛先、件名プレフィックス）
  - フォーム設定（承認者マッピング）
  - 範囲設定（品目情報、日付、チェックボックス範囲）
  - エラーメッセージ
- **運用**: 年度変更時やメールアドレス変更時にこのファイルのみ修正

#### commonLib.gs (共通ライブラリ)
- **目的**: 重複コードを排除し、保守性を向上
- **主な関数**:
  - `generateReferenceUrl()`: スプレッドシート参照URL生成
  - `getOrderItem(row, sheet)`: 発注品目情報取得
  - `generateApprovalToPurchaseBody()`: 承認→購買依頼メール本文生成
  - `generatePurchaseToManagerBody()`: 購買→管理者メール本文生成
  - `sendEmail()`: エラーハンドリング付きメール送信
  - `protectRow()` / `unprotectRow()`: 行保護/解除
  - `logError()`: エラーログ記録
  - `getTimestamp()`: タイムスタンプ取得

#### ApprovalFunctionR61_pjA.gs (メイン承認ワークフロー)
- **目的**: スプレッドシート上の承認チェックボックス変更を処理
- **トリガー**: `onEdit` (スプレッドシート編集時自動実行)
- **備考**: 実行時間上限チェックを追加し、長時間実行時に安全に中断できるようにしています。
- **処理フロー**:
  1. 部署承認チェック → 購買担当者へメール送信
  2. 購買処理チェック → 管理者へメール送信
  3. 最終承認チェック → 行保護
  4. 最終承認解除 → 行保護解除

#### initSheetR1_pjA.gs (シート初期化)
- **目的**: 承認欄のアクセス権保護
- **トリガー**: `onOpen` (スプレッドシート開時自動実行)
- **処理内容**: 承認チェックボックス範囲(S2:S1000)の保護設定

#### initSheet_pjA.gs (シート初期化)
- **目的**: 承認欄のアクセス権保護
- **トリガー**: `onOpen` (スプレッドシート開時自動実行)
- **処理内容**: 承認チェックボックス範囲(S2:S1000)の保護設定

#### sendFormR2_pjA.gs (フォーム送信処理)
- **目的**: Google Forms送信時の承認依頼メール送信
- **トリガー**: フォーム送信イベント
- **処理内容**:
  - フォーム回答から申請者情報、所属課を取得
  - 所属課に応じた承認者へメール送信
  - 承認スプレッドシートURLを本文に記載

### 運用方法

#### 通常運用
1. **フォーム送信**: 申請者がGoogle Formsから申請
2. **承認依頼メール**: 自動で課長へ承認依頼メール送信
3. **部署承認**: 課長がスプレッドシートでチェックボックスON
4. **購買依頼メール**: 自動で購買担当者へ依頼メール送信
5. **購買処理**: 購買担当者がチェックボックスON
6. **発注確定メール**: 自動で管理者へ確定メール送信
7. **最終承認**: 管理者がチェックボックスONで処理完了（行保護）

#### 設定変更時
- **メールアドレス変更**: `config.gs` の `EMAIL_CONFIG` 修正
- **承認者変更**: `config.gs` の `FORM_CONFIG.APPROVERS` 修正
- **年度変更**: `config.gs` の `SPREADSHEET_CONFIG` 修正

#### エラーハンドリング
- 全ての関数でtry-catchを実装
- エラー発生時は `console.error` でログ記録
- メール送信エラー時はアラート表示
- 実行時間超過時の中断ロジックを追加し、スクリプト実行制限への対策を強化

### 保守性向上ポイント
- **設定一元化**: 変更が `config.gs` に集中
- **共通化**: 重複コードを `commonLib.gs` に集約
- **命名統一**: 関数名、変数名を明確に統一
- **エラー処理統一**: 全関数で一貫したエラーハンドリング

## 🔄 リファクタリング詳細
### 背景
- 元々1つのファイル（ApprovalFunctionR50_pjA.gs）で全ての機能を管理していたが、保守性が低下。
- 設定値のハードコーディングが多く、変更時の影響範囲が広かった。
- エラー処理が不統一で、デバッグが困難。

### 実施内容
- **ファイル分割**: 機能を5つのファイルに分割（config.gs, commonLib.gs, ApprovalFunctionR61_pjA.gs, initSheetR1_pjA.gs, sendFormR2_pjA.gs）。
- **設定一元化**: config.gs で全設定を管理、ハードコーディングを排除。
- **共通ライブラリ化**: commonLib.gs で重複関数を統合。
- **エラー処理統一**: 全関数に try-catch を適用、一貫したログ記録。
- **命名改善**: 関数名、変数名を明確に統一。
- **実行時間上限チェック追加**: `ApprovalFunctionR61_pjA.gs` に実行時間監視と中断ロジックを追加。
- **ファイル名更新**: バージョン番号を明示的に付与（initSheetR1_pjA.gs, sendFormR2_pjA.gs）。

### 利点
- **保守性向上**: 変更が局所化され、影響範囲が予測可能。
- **再利用性**: 共通関数を他のプロジェクトで活用可能。
- **デバッグ効率**: エラーログの一元化で問題特定が容易。
- **拡張性**: 新機能追加時の影響を最小限に抑制。

### 影響範囲
- スプレッドシートID、メールアドレスなどの設定変更が config.gs のみで対応可能。
- 年度変更時のURL修正が容易。
- コードの可読性が向上し、新規開発者の参入障壁が低下。
